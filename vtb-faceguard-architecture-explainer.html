VTB FaceGuard

Демонстрационный сервис дополнительной защиты карты: клиент добровольно подключает биометрическую проверку, сохраняет эталонное фото, а затем при снятии наличных система сопоставляет лицо у банкомата с эталоном, проверяет liveness и выбирает понятный продуктовый сценарий.

Снижает риск снятия наличных по украденной карте Добавляет второй фактор проверки именно в момент самой чувствительной операции.
Объяснимое поведение системы Вместо «чёрного ящика» продукт показывает понятные шаги: detection, liveness, verification, decision.
Подходит для demo и пилотной эволюции MVP уже демонстрирует core-механику и при этом оставляет ясный путь к более зрелой версии.
Core decision allow / ask_confirmation / delay_and_alert Решение опирается на match score, liveness signal и читаемые product thresholds.
MVP approach FastAPI + React + facenet-pytorch Локальный CPU-friendly пайплайн без сложной инфраструктуры и без тяжёлых внешних зависимостей.

Core feature flow

Сквозной продуктовый сценарий от подключения защиты до решения банкомата. Здесь важно не только распознавание лица, но и понятная оркестрация шагов.

01

Подключение защиты

В интерфейсе, похожем на ВТБ Онлайн, клиент включает дополнительную защиту снятия наличных.

02

Загрузка эталонного фото

Клиент загружает фото лица. Backend строит embedding и сохраняет связку фото + вектор профиля.

03

Симуляция банкомата

На ATM-экране камера ноутбука играет роль камеры банкомата. Пользователь «прикладывает карту» и запускает проверку.

04

Получение кадра

Frontend получает текущий frame или fallback image и отправляет его на backend вместе с profile_id.

05

Face detection

Из кадра выделяется лицо. Если лицо не найдено или в кадре несколько лиц, система сразу возвращает понятный reason.

06

Liveness detection

Challenge-response слой просит повернуть голову и оценивает изменение геометрии landmarks между кадрами.

07

Face verification

Embedding текущего лица сравнивается с эталонным embedding профиля через cosine similarity.

08

Decision engine

Match score, liveness status и специальные флаги превращаются в продуктовый итог: allow, ask_confirmation или delay_and_alert.

Models and algorithms

MVP построен на понятных и объяснимых CV-подходах. Цель здесь не «магия AI», а инженерно стабильная демонстрация product logic.

Face detection

MTCNN

Используется для поиска лица на изображении и получения facial landmarks. Это даёт базовый quality gate: нет лица, несколько лиц, неудачный кадр.

Face embedding

InceptionResnetV1

Из cropped face строится embedding-вектор. Именно он используется как компактное представление лица для сравнения с эталоном.

Liveness MVP

Challenge-response + landmarks

Система просит слегка повернуть голову и оценивает, изменяется ли геометрия лица естественным образом, а не просто сдвигается картинка.

Как работает face verification

Эталонное фото пользователя и текущий кадр преобразуются в embeddings. Затем считается cosine similarity: чем ближе векторы, тем выше вероятность, что перед камерой тот же человек.

similarity = cosine(embedding_current, embedding_reference) if similarity >= MATCH_ALLOW: decision_candidate = "allow" elif similarity >= MATCH_REVIEW: decision_candidate = "ask_confirmation" else: decision_candidate = "delay_and_alert"

Как формируется confidence

В MVP confidence не претендует на строгую статистическую вероятность. Это нормализованная presentation metric, которая помогает на экране показать итог проверки в удобной для пользователя форме.

raw similarity -> threshold bands -> normalized confidence below review threshold -> low confidence between review and allow -> medium confidence above allow threshold -> high confidence

Почему такой подход выбран для MVP

  • подходит для локального demo без тяжёлой инфраструктуры
  • работает на CPU и не требует отдельного GPU-контура
  • объясним на защите: detection → embedding → similarity → decision
  • позволяет вручную крутить thresholds перед demo

Как работает liveness в текущем MVP

  • берутся два кадра: базовый и challenge frame
  • сравниваются landmarks лица и сдвиг bounding box
  • простое движение телефона не считается хорошим сигналом
  • при провале возможен мягкий retry до финального red scenario

Technical architecture

Архитектура намеренно компактная: frontend orchestration, backend API, ML-layer, локальное storage и product-facing decision layer.

Client side

VTB-style frontend

React/Vite интерфейс имитирует mobile-bank flow: dashboard, protection setup, ATM demo page.

Camera input

Используется live camera stream или fallback image. Frontend собирает кадр и вызывает backend endpoint.

Decision rendering

На ATM page отображаются status panel, confirmation flow, event timeline и решение системы.

Backend + ML side

FastAPI API layer

Endpoints для enroll и ATM check. Принимают файлы, profile_id и возвращают человекочитаемый decision payload.

Face engine

MTCNN детектирует лицо и landmarks, InceptionResnetV1 строит embedding, cosine similarity оценивает совпадение.

Liveness service

Challenge-response слой сравнивает landmarks между кадрами и выдаёт liveness score/status.

Decision engine

Один явный модуль объединяет similarity, liveness и flags в green/yellow/red product scenarios.

Storage

Локальное JSON storage хранит профили, пути к фото и эталонные embeddings для demo/MVP режима.

Архитектурно решение специально сделано «плоским» и читаемым: это упрощает защиту проекта, ручную калибровку и дальнейший переход от demo к пилоту.

End-to-end pipeline

Ниже показан полный data flow: от загрузки фото до финального решения на ATM-экране.

Enroll flow

Этап подключения защиты и создания эталонного профиля.
1. Upload Клиент загружает фото в интерфейсе защиты.
2. Detection Backend ищет лицо и валидирует кадр.
3. Embedding Из лица извлекается эталонный вектор.
4. Storage Фото, profile_id и embedding сохраняются в локальном storage.

Verification flow

Проверка личности в ATM-сценарии.
1. Card action Пользователь нажимает «Приложить карту».
2. Camera capture Frontend собирает frame и challenge frame.
3. Backend check `/api/atm/check` получает кадры и profile_id.
4. Result payload API возвращает scores, status, decision, flags и event log.

Scoring + decision

Как технические сигналы превращаются в продуктовый outcome.
Match score Cosine similarity между current face и reference face.
Liveness score Signal по challenge-response и landmarks dynamics.
Decision engine Thresholds + rules + quality flags.
UI rendering allow / ask_confirmation / delay_and_alert.

Decision outcomes

Продуктовые сценарии intentionally сведены к трём, чтобы их легко было объяснить и показать как банковую логику защиты.

Green

allow

Лицо совпало с эталоном, liveness подтверждён, банкомат продолжает операцию без задержек.

Yellow

ask_confirmation

Есть частичное несоответствие или спорный score. Система показывает владельцу вопрос «Это вы?».

Red

delay_and_alert

Низкий match, провал liveness или подозрение на spoof. Операция задерживается и запускается защитный сценарий.

MVP limitations

Это честный demo/MVP. Некоторые вещи намеренно упрощены, чтобы успеть собрать продуктовую механику и убедительный end-to-end сценарий.

Что упрощено

  • локальное JSON storage вместо production-grade profile store
  • challenge-response liveness вместо полноценной passive anti-spoof модели
  • demo-presets и fallback сценарии для стабильной презентации
  • симуляция ATM поведения без реальной интеграции с железом банкомата

Какие риски остаются

  • качество live demo зависит от камеры, света и угла съёмки
  • liveness MVP ещё чувствителен к условиям и нуждается в калибровке
  • thresholds подобраны для demo и пилота, а не для масштабного production rollout
  • нужны более строгие требования к безопасности, данным и аудиту

Next evolution

После demo MVP продукт естественно развивается в более зрелую банковую систему защиты.

Следующий продуктовый слой

  • trusted persons / доверенные лица
  • история событий и журнал проверок в приложении
  • push-подтверждение вместо только локального ATM prompt
  • реальные банковские сценарии блокировки, задержки, escalation

Следующий технический слой

  • более сильная anti-spoof модель
  • настоящая интеграция с ATM systems и банковыми сервисами
  • risk scoring с учётом контекста операции
  • production storage, observability, policy engine и управление моделями

# VTB FaceGuard — Project Context

## 1. Обзор проекта

### Название продукта
**VTB FaceGuard**  
Рабочее название demo/MVP сервиса дополнительной защиты снятия наличных.

### Краткое описание
VTB FaceGuard — это сервис, который добавляет биометрическую и поведенческую проверку к операциям снятия наличных. Клиент добровольно подключает защиту в интерфейсе, похожем на ВТБ Онлайн, сохраняет эталонное фото лица, а затем при операции у банкомата система сверяет лицо перед камерой с эталоном, проверяет liveness и выбирает продуктовый сценарий реакции.

### Основная ценность для банка и клиента
- Для банка:
  - снижение риска снятия наличных по украденной или скомпрометированной карте;
  - дополнительный антифрод-слой в точке выдачи денег;
  - возможность перейти от реактивной защиты к превентивной.
- Для клиента:
  - дополнительная защита средств;
  - более уверенное использование карты в офлайн-сценариях;
  - понятная логика поведения системы в спорных ситуациях.

### Целевая аудитория и сценарии использования
- Розничные клиенты банка, которые хотят подключить дополнительную защиту.
- Команды антифрода и безопасности банка.
- Продуктовые и технические команды, которые оценивают потенциал пилота.
- Demo-сценарии для жюри, стейкхолдеров, внутренних презентаций и пилотных обсуждений.

Основной сценарий использования:
- клиент подключает защиту;
- сохраняет эталонное фото;
- приходит к банкомату;
- система анализирует лицо и поведение у банкомата;
- операция либо проходит без трения, либо требует подтверждения, либо задерживается как подозрительная.

## 2. Проблема и решение

### Проблема
Классический сценарий “карта + PIN” уязвим в случаях:
- кражи карты;
- компрометации PIN-кода;
- социального инжиниринга;
- принуждения;
- использования карты третьим лицом без явного цифрового следа.

В ATM-сценарии особенно критично то, что операция физически происходит в точке выдачи денег, и если защита сработала слишком поздно, ущерб уже нанесён.

### Что решает продукт
Продукт добавляет дополнительный слой проверки личности и поведения в момент снятия наличных:
- проверяет, тот ли человек стоит перед банкоматом;
- анализирует, выглядит ли этот человек “живым”, а не как фото/экран;
- в перспективе учитывает поведенческие и видеоаномалии вокруг банкомата.

### Чем решение отличается от “просто PIN + карта”
- PIN подтверждает знание секрета, но не подтверждает личность человека у банкомата.
- FaceGuard добавляет биометрическую привязку к владельцу карты.
- При спорном результате система не обязательно блокирует всё сразу: она может перейти в step-up сценарий и запросить подтверждение.

### Формулировка технологии
Ключевая технологическая идея проекта:

**анализ потокового видеоряда для распознавания действий и выявления отклонений от стандартного поведения**

В текущем MVP эта идея реализована в упрощённом виде:
- анализ отдельных кадров и короткого challenge-response сценария;
- оценка совпадения лица;
- базовая liveness-проверка;
- подготовка архитектуры под более глубокую видеоаналитику и поведенческий анализ.

## 3. Core-фича

### Пошаговый сценарий

#### Шаг 1. Подключение защиты
Клиент открывает экран `ВТБ Защита` и включает дополнительную защиту снятия наличных.

Используемые данные:
- состояние услуги (`inactive / pending / active`);
- локальный статус подключения защиты.

#### Шаг 2. Загрузка эталонного фото
Клиент сохраняет фото лица:
- либо загружает файл;
- либо делает фото прямо с камеры устройства.

Используемые данные:
- изображение лица;
- метаданные файла;
- локальный preview для UI.

#### Шаг 3. Построение шаблона
Backend получает изображение и:
- ищет лицо;
- извлекает embedding;
- сохраняет профиль с `profile_id`, фото и embedding.

Используемые данные:
- raw image bytes;
- embedding-вектор;
- путь к сохранённому фото;
- `profile_id`.

#### Шаг 4. Работа на банкомате
В demo-режиме камера ноутбука играет роль камеры банкомата. Пользователь нажимает `Приложить карту`, и начинается проверка.

Используемые данные:
- текущий кадр с камеры или fallback image;
- `profile_id` активного клиента;
- при challenge-response — второй кадр.

#### Шаг 5. Face detection
Backend проверяет:
- есть ли лицо в кадре;
- одно ли лицо в кадре;
- достаточно ли хорошее качество кадра.

Если лицо не найдено или лиц несколько, система сразу возвращает reason и понятный сценарий для UI.

#### Шаг 6. Liveness detection
Система просит слегка повернуть голову и сравнивает базовый кадр с challenge-кадром.

Используемые данные:
- bounding box лица;
- landmarks;
- изменение геометрии лица между кадрами.

#### Шаг 7. Face verification
Embedding текущего лица сравнивается с эталонным embedding профиля.

Используемые данные:
- current embedding;
- reference embedding;
- similarity score.

#### Шаг 8. Решение
Decision engine объединяет:
- match score;
- liveness score/status;
- quality flags;
- product thresholds.

И возвращает один из трёх сценариев:
- `allow`
- `ask_confirmation`
- `delay_and_alert`

### Взаимодействие клиента с системой

#### В мобильном / банковом интерфейсе
Клиент:
- включает защиту;
- загружает фото;
- получает подтверждение, что защита активна.

#### В ATM-сценарии
Клиент:
- инициирует операцию;
- смотрит в камеру;
- в спорном случае подтверждает “Это вы?”;
- при red-сценарии видит, что операция задержана системой защиты.

## 4. Видеоаналитика и поведение

### Что уже заложено
Сейчас MVP реально анализирует:
- наличие лица в кадре;
- количество лиц;
- базовый challenge-response для liveness.

### Какие поведенческие / видео-фичи заложены как направление развития

#### Второй человек в кадре
Возможный сценарий:
- рядом с клиентом находится ещё один человек;
- это может быть признаком shoulder surfing, давления или вмешательства.

Потенциальная реакция:
- yellow или red signal;
- повышение risk score;
- рекомендация задержать операцию или запросить подтверждение.

#### Loitering / слишком долгое нахождение у банкомата
Возможный сценарий:
- человек слишком долго находится у банкомата без естественного паттерна действий;
- это может быть подозрительным поведением перед атакой или вмешательством.

Потенциальная реакция:
- накопление поведенческого риска;
- escalation к дополнительной проверке.

#### Попытка перекрыть камеру
Возможный сценарий:
- камера закрывается рукой, предметом или намеренно уводится от лица;
- качество видеопотока резко деградирует.

Потенциальная реакция:
- флаг вмешательства;
- red scenario;
- логирование инцидента.

#### Вмешательство в устройство
Возможный сценарий:
- нестандартные действия у банкомата;
- подозрительное нахождение в сервисной зоне;
- перекрытие обзора, попытки манипуляции.

Потенциальная реакция:
- поведенческий alert;
- запись события;
- эскалация в anti-fraud / security monitoring.

### Связь с формулировкой про потоковый видеоряд
Фраза “анализ потокового видеоряда для распознавания действий и выявления отклонений от стандартного поведения” относится не только к распознаванию лица, но и к следующему слою системы:
- отслеживание длительности и контекста действий;
- анализ присутствия других людей;
- распознавание нехарактерных паттернов поведения;
- оценка отклонений от “нормальной” траектории взаимодействия с банкоматом.

Таким образом, face verification — это только одна из частей более общей системы видеоаналитики.

## 5. Модели и алгоритмы

### Используемые модели и подходы

#### Face detection
Назначение:
- найти лицо на изображении;
- определить, сколько лиц в кадре;
- получить landmarks для последующей обработки.

В текущем MVP:
- используется `MTCNN`.

#### Face verification
Назначение:
- сравнить лицо у банкомата с эталонным лицом клиента.

В текущем MVP:
- из лица строится embedding;
- embeddings сравниваются через cosine similarity.

#### Liveness / anti-spoof
Назначение:
- проверить, что перед камерой находится живой человек, а не фото или replay.

В текущем MVP:
- используется challenge-response;
- оценивается изменение landmarks между кадрами.

#### Video-based anomaly / behavior detection
Назначение:
- распознавать отклонения от нормального поведения у банкомата.

Статус:
- пока не реализовано как полноценный модуль;
- заложено в продуктовое позиционирование и архитектурное развитие.

### Для чего нужна каждая модель
- Face detection:
  - выделение лица;
  - quality gate;
  - multiple faces detection.
- Face verification:
  - проверка совпадения с владельцем профиля.
- Liveness:
  - защита от фото, экрана, replay attack.
- Behavior / anomaly layer:
  - расширение антифрод-логики за пределы простого сравнения лица.

### Как работает общий пайплайн

1. Frontend получает изображение или кадры с камеры.
2. Backend принимает `file` и `profile_id`.
3. Face detection находит лицо и landmarks.
4. Face embedding извлекает вектор признаков.
5. Similarity сравнивает текущий embedding с эталоном.
6. Liveness анализирует challenge-кадры.
7. Decision engine объединяет сигналы и thresholds.
8. Frontend получает product-friendly ответ и рендерит сценарий.

## 6. Техническая архитектура

### Общая архитектура системы

Система состоит из следующих основных компонентов:

- Frontend:
  - интерфейс, похожий на ВТБ Онлайн;
  - ATM demo screen;
  - orchestration пользовательских flow.

- Backend API:
  - приём файлов;
  - API для enroll и ATM check;
  - orchestration ML/decision pipeline.

- CV/ML layer:
  - face detection;
  - embedding extraction;
  - similarity scoring;
  - liveness service.

- Decision layer:
  - product rules;
  - thresholds;
  - финальный decision payload.

- Storage:
  - локальное JSON-хранилище профилей;
  - сохранённые фото;
  - embeddings.

- ATM / camera input:
  - в demo используется камера ноутбука;
  - в будущем предполагается реальный ATM camera feed и события банкомата.

### Высокоуровневое взаимодействие компонентов

1. Frontend инициирует enroll или verification.
2. Backend API принимает запрос.
3. ML layer анализирует кадры.
4. Decision engine превращает ML-сигналы в продуктовое решение.
5. Backend возвращает structured response во frontend.
6. Frontend отображает результат и event timeline.

### Какие данные куда ходят

#### В enroll flow
- Frontend → Backend:
  - файл с эталонным фото.
- Backend → Storage:
  - фото;
  - embedding;
  - `profile_id`.
- Backend → Frontend:
  - `profile_id`;
  - сообщение об успехе;
  - метаданные профиля.

#### В ATM verification flow
- Frontend → Backend:
  - основной кадр;
  - optional challenge frame;
  - `profile_id`.
- Backend → Storage:
  - чтение эталонного embedding профиля.
- Backend → Frontend:
  - similarity score;
  - normalized confidence;
  - liveness score/status;
  - decision;
  - flags;
  - event log.

### Взаимодействие с банкоматом
В текущем MVP:
- банкомат симулируется отдельным экраном frontend;
- камера ноутбука играет роль камеры банкомата;
- действие `Приложить карту` моделирует начало ATM-операции.

В будущей версии предполагается, что банкомат или ATM middleware присылает:
- фото или короткий видеосегмент;
- события начала операции;
- контекст операции;
- сигналы об устройстве и состоянии сессии.

### Как принимается итоговое решение
Итоговое решение принимает `decision engine`, который объединяет:
- результат face verification;
- liveness;
- служебные флаги;
- quality conditions;
- thresholds из конфигурации.

Результат отдается:
- во frontend ATM UI;
- в будущем — в банковые сервисы, anti-fraud контур, уведомления и журнал событий.

## 8. API и основные эндпоинты

### Реально используемые эндпоинты

#### `POST /api/enroll`
Назначение:
- регистрация эталонного фото клиента.

Принимает:
- `multipart/form-data`
- поле `file`

Возвращает:
- `success`
- `profile_id`
- `photo_path`
- `message`
- `face_confidence`
- `embedding_size`

#### `POST /api/atm/check`
Назначение:
- проверка лица и liveness в ATM-сценарии.

Принимает:
- `multipart/form-data`
- `file`
- optional `challenge_file`
- `profile_id`

Возвращает:
- `decision`
- `title`
- `message`
- `severity`
- `recommended_ui_action`
- `similarity_score`
- `normalized_confidence`
- `liveness_score`
- `liveness_status`
- `event_log`
- `flags`

#### `GET /api/health`
Назначение:
- health-check backend-сервиса.

### Частично подготовленные / legacy эндпоинты

#### `POST /api/face/register`
Статус:
- заготовка / не является ядром текущего flow.

#### `POST /api/face/verify`
Статус:
- заготовка / не используется как основной ATM pipeline.

### Логика уровней API
- Enroll API отвечает за создание эталонного профиля.
- ATM Check API отвечает за верификацию и decision payload.
- В будущем могут появиться:
  - API включения/отключения защиты;
  - API истории событий;
  - API подтверждения владельца;
  - API банковых уведомлений и anti-fraud signals.

## 9. Ограничения и допущения MVP

### Что сейчас упрощено
- `enableProtection` / `disableProtection` на frontend пока mock;
- storage локальный, на JSON-файле;
- ATM в текущем виде — это frontend demo, а не реальное ATM integration;
- часть legacy экранов и заготовок всё ещё лежит в проекте;
- liveness реализован как MVP-эвристика, а не production anti-spoof.

### Ограничения по качеству и инфраструктуре
- точность зависит от камеры, освещения и качества кадра;
- пороги сейчас подобраны для demo/pilot логики;
- нет полноценного production logging / audit / secure storage;
- нет реальных банковых интеграций и распределённой инфраструктуры;
- нет отдельного ML serving контура.

### Что важно понимать другому агенту
- не стоит воспринимать текущий liveness как финальный production-ready anti-spoof;
- часть product story уже шире текущей реализации и включает будущую видеоаналитику;
- demo presets — это не баг, а осознанный инструмент стабильной презентации;
- проект пригоден для demo, архитектурного обсуждения и пилотного прототипирования, но не для немедленного production rollout.

- углубление видеоаналитики;
